## The reframe (do this first): one decision per page, per field
Today a page can have dozens of candidate cards (homepage = 261, /for-businesses/ = 33 incl. 16 title + 16 meta variants, mostly rejected). Re-frame the surface so the default view is **one current value + one recommended proposal per (page, field)**:
- Default: show the **single highest-scoring active (pending/approved) proposal** per `(page, mutation_type/field)`.
- **Collapse alternatives** behind a "See N alternatives" control; **hide `rejected` by default** behind a "Show rejected" toggle.
- Show clearly **which proposal is live/approved** for that field.

### A2 — Action Card must show ONE action (highest priority)
`/dashboard/seo/action/[id]` currently dumps the page's *entire* candidate list (32 cards for for-businesses; 261 for homepage). It must render **only the single candidate `[id]`** (its current value, proposed value, evidence, lane, measurement plan). Page-level history/alternatives belong in a collapsed "alternatives" section, not the default.

### A3 — Reflect one-active-approved per field
A page has one title and one meta, but the data has 3 approved titles + 3 approved metas for for-businesses. In the UI: surface only the **single active/winning approved proposal** per field (use the back-end's active flag once available; until then pick highest score among approved) and label it "live." Don't render multiple conflicting "approved" cards as equals.

### A4/A1 — De-clutter
Hide rejected by default; collapse the variant pile to one recommended + "see alternatives."

### A5/A6 — Labels & copy
Add a legend/tooltip for the `hybrid`/`auto` tags (or remove them from the primary view). Replace the repeated boilerplate ("Update title_tag_refinement on … 14297 impressions at position 75.8" on every variant) with the **differentiating** info — the proposed text + a one-line "why this one."
