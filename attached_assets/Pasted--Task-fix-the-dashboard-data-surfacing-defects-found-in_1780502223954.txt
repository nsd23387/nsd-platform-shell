## 1. HIGH — Competitors screen reads ~2% of its data

The proxy route `/api/proxy/seo/competitor-gaps` itself returns only 10 rows / 1 competitor (a backend `LIMIT 10` + implicit single-competitor scoping) — confirmed by calling it directly, so the cap is in the route, not the component.

**Fix (`app/api/proxy/seo/competitor-gaps/route.ts` + the Competitors page):**
- Remove the `LIMIT 10` and any single-competitor `WHERE`. Return the full governed feed (all 9 competitors, both `gap_type`s), ordered by `opportunity_score DESC`. Paginate if needed (e.g. server-side page size 50 + "load more"), but the KPI counts must reflect the **full** table.
- Fix the KPI tiles to count the whole table: **Total gaps = 483, Competitors tracked = 9, Keyword gaps = 469, Content gaps = 14** (compute from the data, don't hardcode). "Content gaps 0" is currently flatly wrong.
- Populate the **Volume / Difficulty** columns (currently all "—") by joining each gap `keyword` to `analytics.external_query_intelligence` on normalized query (`search_volume`, `keyword_difficulty`, `cpc`); leave "—" only when genuinely absent.
- Columns: `competitor_url, gap_type, keyword, competitor_ranking_position, our_ranking_position, competitor_page_url, opportunity_score, status`.

**Out of scope (note, don't build):** the competitor *set* is only 9 and DataForSEO competitor enrichment is thin — expanding the tracked competitor list / ingesting `metrics_ahrefs_keyword_gap` (5,960) + `authority_gap_intelligence` (3,913) is a separate backend ingest task. This fix is to surface the full governed feed that already exists.

**Acceptance:** Competitors KPIs read 483 / 9 / 469 / 14; the table lists gaps across all 9 competitors; volume/difficulty populated where `external_query_intelligence` has the keyword; the proxy returns the full set (paginated), verified `count == 483`.

## 2. MEDIUM — one GSC metric, three different numbers across screens
**Symptom:** "30-day organic clicks" shows as **9** (Command Center hero), **10** (Command Center Momentum), **7** (Results). Same for impressions/position. Cause: each surface computes from a different grain (date-only `metrics_search_console_daily` vs page-level `metrics_search_console_page_daily` vs query-level) and/or a different window anchor.

**Fix — a single shared metrics module (`lib/seoMetrics.ts`) every surface reads:**
- `organicClicks(windowDays)` and `organicImpressions(windowDays)` = sum over **`analytics.metrics_search_console_daily`** (the complete date-only grain — most accurate; page/query grains undercount), window = `metric_date > (max(metric_date) - windowDays)` (anchor to the table's max date, since GSC lags ~4 days — do NOT anchor to `CURRENT_DATE`, which silently drops the lag days).
- `avgPosition(windowDays)` = impression-weighted avg position over the same grain/window.
- `topQueryPosition(pagePath)` = the highest-impression query's avg position from `metrics_search_console_query_page_daily` (the same `DISTINCT ON (path) … ORDER BY impressions DESC` used by `/portfolio`).
- Point the **Command Center hero, Command Center Momentum, and Results** cards at these functions. Result: one "30-day clicks" number dashboard-wide (fixes the hero 9-vs-10 off-by-one too).

**Acceptance:** Command Center hero == Momentum == Results for the same window (e.g. 30-day clicks identical on all three); each equals the `metrics_search_console_daily` sum over the max-date-anchored window.
